# Go House Style uses multi-context vertical modules

Go services organize bounded contexts as sibling modules under `internal/business/<context>`. Every context has `application/application.go`, `application/assembler.go`, and a context-named Fx module; Domain, command, query, event-handler, Transport, and Infrastructure packages exist only when the context owns those responsibilities. Application DTO/Domain Entity mapping lives in `application/assembler.go`, while a context with persistence uses `infrastructure/convert.go` for Data Object/Domain Entity mapping. Generated stubs live under `gen/`, matching contract sources under `proto/`. Shared technical runtimes live under `internal/pkg`, tests stay beside their packages, and conditional mechanisms do not pre-create empty directories in the baseline tree.
