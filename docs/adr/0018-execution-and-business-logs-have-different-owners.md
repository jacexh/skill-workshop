# Execution and business logs have different owners

Transport or Runtime records one Execution Completion Log for each inbound request, Integration Message, task, scheduled trigger, or owned runtime operation, including end-to-end duration and protocol/delivery outcome. Application Commands and Queries do not repeat that completion record; they may emit a Business Semantic Log only when a decision or fact has independent operational value, while durable business evidence remains a Domain Event, audit record, or persisted state. Application becomes the execution logger only when it owns a terminal flow with no outer observer, such as a Domain Event Handler whose contract cannot return an error, or when it deliberately suppresses a best-effort failure. Domain never logs, and Infrastructure enriches and returns errors unless it owns retry, suppression, or a terminal runtime operation. Provider errors receive an oops stack and owned context when they first enter controlled code; later layers add context only for new semantics, preserve `errors.Is/As`, and never mechanically log-and-return the same error. Public error mapping preserves the internal cause for the single outer logger. Logs use `sloghelper.Error`, classify expected rejection separately from internal failure, and never expose secrets, sensitive payloads, or an unredacted resolved configuration.
