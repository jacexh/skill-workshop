# Saved Aggregate instances are stale

A new persisted Aggregate starts with in-memory `Version == 0` and is inserted with stored version `1`; an existing Aggregate update compares its loaded version, increments the stored version atomically, and maps an affected-row mismatch to a stable concurrency conflict. Repository `Save` does not refresh the in-memory version. After a successful Save, callers may read existing result fields, assemble a DTO, and drain the Domain Events already recorded by that transaction, but they may not expose the stale Version as the newly persisted concurrency token, invoke further business mutations, or save the same instance again. A caller that requires the current token reloads it. A later transaction reloads a fresh Aggregate and `event.Collection`. This contract matches go-jimu's one-shot event collection, whose drained instance cannot accept another event batch, and must be protected by Repository integration tests for insert versioning and stale-version conflicts.
