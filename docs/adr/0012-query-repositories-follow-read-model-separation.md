# Query Repositories follow read-model separation

A query name or external endpoint does not by itself require a QueryRepository. Command-side Aggregate loading always uses the Domain Repository, and a Focused Aggregate Read may also load one Aggregate there when full reconstitution is reasonable and no distinct read semantics or Aggregate API pollution results. Lists, pages, history, reports, statistics, cross-Aggregate composition, denormalized views, partial optimized reads, projections, or independently governed freshness, authorization, source, and performance use an Application-owned QueryRepository returning transport- and storage-neutral read models. Infrastructure implements that port, while Transport always delegates through an Application Query Handler and never calls either Repository directly.
