# Code Agent 可维护性文章设计

- **状态**: Draft
- **日期**: 2026-04-27
- **作者**: xuhao + Codex
- **产物类型**: 公众号文章
- **工作主题**: 反驳并重构“AI 能写代码，但写不出人类可以维护的代码”这一判断

## 背景

文章起因是一个常见判断:AI 能写代码,但写不出人类可以维护的代码。

这个判断有现实来源。很多人用 AI 写出来的代码确实像一次性脚本:能跑,但缺少上下文意识、架构边界、测试反馈、团队约定和长期演进能力。但本文不把问题归因于“AI 天然写不出可维护代码”,而是指出一个更关键的误用:

> 大多数人只调用了 code agent 的“实现”能力,却期待它交付完整工程质量。

本文要把讨论从“AI 会不会写代码”升级为“怎样把 code agent 放进可维护的工程闭环”。

## 目标

1. 提出作者的独到判断:code agent 的核心价值不是替人敲代码,而是参与需求、设计、实现、验证、审查、记忆的完整工程闭环。
2. 纠正常见误解:“AI 写不出可维护代码”很多时候是错误归因,真正缺失的是上下文、约束、反馈和沉淀。
3. 给困惑读者启发:不要只研究 prompt,而要设计 agent 工作流。
4. 以中等露出方式介绍 `skill-workshop`,把它作为作者正在实践的方法论证据,而不是把文章写成项目广告。

## 目标读者

本文面向多类读者,但保持方法论高度:

| 读者 | 他们关心的问题 | 文章需要给出的价值 |
|---|---|---|
| 技术管理者 / CTO | AI 编程如何进入工程体系 | 判断框架与组织落地视角 |
| 一线工程师 | 为什么 AI 代码经常不好维护 | 可操作的协作方式与反思角度 |
| 独立开发者 | 怎样从“让 AI 写代码”升级 | code agent 工作流思维 |
| 怀疑者 / 泛技术读者 | AI 编程是不是噱头 | 重新理解“可维护性”的来源 |

## 核心论点

文章的核心主张:

> 真正的问题不是 AI 能不能写出可维护代码,而是我们有没有把 code agent 放进一个可维护的工程闭环里。

展开为三层:

1. 可维护性不是单次编码动作的结果,而是上下文、约束、反馈和沉淀共同塑造的结果。
2. 如果只让 AI 做“实现”,它当然像代码生成器;如果让 agent 进入完整工程闭环,它才开始像工程协作者。
3. AI 编程真正拉开差距的地方,不是谁更会让模型吐代码,而是谁更早把隐性的工程经验沉淀成 agent 可执行的技能、标准、记忆和检查点。

## 主框架

文章使用一张框架图式作为主线:

```text
上下文 -> 标准 -> 计划 -> 实现 -> 验证 -> 审查 -> 记忆
```

各环含义:

| 环节 | 含义 | 缺失后的典型问题 |
|---|---|---|
| 上下文 | 让 agent 理解项目结构、历史决策、约定和当前目标 | 代码局部正确,系统语义错误 |
| 标准 | 注入团队架构规范、编码约束、测试原则 | 风格漂移,边界混乱,不可审查 |
| 计划 | 在动手前拆解任务、识别风险、明确验收标准 | 一边写一边猜,改动范围失控 |
| 实现 | 根据计划修改代码或产出内容 | 只剩这一环时退化成代码生成 |
| 验证 | 运行测试、检查边界条件、确认行为符合意图 | “看起来能跑”,但缺乏证据 |
| 审查 | 用人类和 agent 共同检查可维护性、风险和一致性 | 缺陷被合并进长期代码库 |
| 记忆 | 把新的决策、约定、经验沉淀回知识库和技能 | 每次会话从零开始,错误反复出现 |

## 文章结构

### 1. 开篇:那个常见判断可能问错了问题

从“AI 能写代码,但写不出人类可以维护的代码”切入。先承认这个观察有现实基础,再转折指出:问题不在“AI 写的代码天然不可维护”,而在于我们常常把 code agent 当成代码生成器使用。

### 2. 重新定义 code agent:不是代码补全,而是工程协作者

区分三层:

| 层级 | 特征 |
|---|---|
| LLM 写代码 | 输入一句需求,输出一段代码 |
| AI IDE 辅助 | 在编辑器里补全、解释、改写 |
| Code agent | 读取代码库、理解任务、规划改动、执行验证、接受审查、沉淀记忆 |

强调本文讨论的是第三层。

### 3. 主框架:Code Agent 可维护性闭环

展开 `上下文 -> 标准 -> 计划 -> 实现 -> 验证 -> 审查 -> 记忆`。说明任何一环缺失,都会让 agent 退化成“会写代码但不懂系统”的工具。

### 4. 为什么很多人用不好:只用了“实现”这一环

列出常见误用:

- 只给一句需求,不给项目上下文
- 只要求生成代码,不要求解释设计取舍
- 没有架构标准或团队约束
- 不要求测试和验证证据
- 没有 review 环节
- 不把新的经验沉淀成可复用记忆

这一节承担“给困惑读者启发”的功能。

### 5. 我的实践:把隐性工程经验做成 agent 技能

以中等露出方式介绍 `skill-workshop`。表达重点不是“这是一个项目介绍”,而是“这是我把方法论工程化的实践”。

对应关系:

| 闭环环节 | `skill-workshop` 中的实践 |
|---|---|
| 上下文 / 记忆 | `superpowers-memory` 维护项目知识库,让 agent 不必每次从零理解项目 |
| 标准 | `superpowers-architect` 把架构模式和团队约束作为 agent 工作时的标准 |
| 验证 | `designing-tests` 把测试设计从“补测试”前移到意图和边界分析 |
| 计划 / 审查 | workflow skills 把 brainstorming、planning、implementation、review 串成流程 |

这一节避免过度广告化。`skill-workshop` 是证据,不是主角。

### 6. 结尾:未来的差距不是谁用了 AI

回到作者判断:

> AI 编程的分水岭,不是谁更会让模型吐代码,而是谁更早把自己的工程体系翻译成 agent 能执行的工作流。

结尾可以轻点成熟度模型:

1. 提示词玩家:让 AI 写片段代码。
2. 工具使用者:用 AI IDE 提升局部效率。
3. Agent 协作者:让 agent 参与任务闭环。
4. 工程系统设计者:把团队经验产品化为技能、标准、记忆和检查点。

## 关键表达

可作为文章中的记忆点:

- “说 AI 写不出可维护代码,很多时候像是在说:一个没有需求文档、没有代码规范、没有测试、没有 review 的初级工程师,写不出可维护代码。”
- “可维护性从来不是某个程序员单次编码动作的结果,而是上下文、约束、反馈和沉淀共同塑造出来的结果。”
- “如果只让 AI 做‘实现’,它当然像一个代码生成器;如果让 agent 进入完整工程闭环,它才开始像一个工程协作者。”
- “我做 `skill-workshop` 的目的,不是收集更多 prompt,而是把工程团队里那些隐性的协作经验,变成 code agent 可以调用、可以遵守、可以复用的技能。”
- “AI 编程的分水岭,不是谁更会让模型吐代码,而是谁更早把自己的工程体系翻译成 agent 能执行的工作流。”

## `skill-workshop` 露出策略

露出强度:中等。

文章主体保持方法论视角,不在开头直接宣传项目。`skill-workshop` 在第 5 节作为实践案例集中出现,并可在第 3 节解释框架时轻微提及。

避免的写法:

- 不把标题写成 `skill-workshop` 项目介绍。
- 不逐项罗列功能参数。
- 不把读者导向“装一个工具就能解决问题”的误解。

推荐的写法:

- 强调它是在把“会用 code agent”工程化。
- 强调 memory、architect、testing、workflow 分别对应闭环中的关键能力。
- 强调工具背后的思想:把隐性经验变成 agent 能执行的协议。

## 非目标

- 不写成模型评测文章。
- 不讨论具体模型排行榜或 IDE 对比。
- 不写成 prompt 技巧合集。
- 不写成纯项目发布稿。
- 不承诺 AI 可以自动替代工程师的判断。

## 成功标准

文章完成后应满足:

1. 读者能复述主论点:可维护性来自工程闭环,不是来自“人写”或“AI 写”的身份标签。
2. 技术管理者能看到组织落地框架。
3. 工程师和独立开发者能获得具体启发:把 agent 带入上下文、标准、验证、审查、记忆。
4. `skill-workshop` 的出现自然,像实践证据,不像硬广。
5. 文章有至少 3 个可被摘录传播的强观点句。
