# Conventions

## Coding Standards

- **Hook runtime (Claude memory):** All hook logic lives in `plugins/superpowers-memory/hooks/hook-runtime.js`. Bash scripts (`pre-tool-use`, `session-start`, `user-prompt-expansion`) are 2-5 line wrappers that `exec node hook-runtime.js <mode>`. JSON output: `hookSpecificOutput` wrapper in plugin env, flat `additional_context` otherwise.
- **Hook runtime (Codex memory):** `codex-plugins/superpowers-memory/hooks/codex-runtime.js` — same modes minus `user-prompt-expansion`, plus `user-prompt-submit`. Runtime-local paths use `path.dirname(__filename)` and skill command examples use `${PLUGIN_ROOT:-codex-plugins/superpowers-memory}`; Codex plugin sources must not rely on Claude compatibility variables. Advisory output uses `hookSpecificOutput.additionalContext`; KB write denial uses `hookSpecificOutput.permissionDecision = "deny"` with `permissionDecisionReason`.
- **Hook scripts (architect Claude):** `superpowers-architect` uses its restored v1.13.10 PreToolUse dynamic pattern loader. Keep hook output progressive: inject compact pattern index/gate guidance, then let the standards workflow load relevant full pattern files on demand.
- **Hook scripts (architect Codex):** `superpowers-architect` uses restored v1.13.10 SessionStart and UserPromptSubmit hooks. SessionStart stays lightweight; UserPromptSubmit triggers only on explicit upstream `$superpowers:*` workflow skill mentions; Stop remains a no-op.
- **Hook scripts (designing-tests Claude):** Bash with `set -euo pipefail`. Hooks stay restrained: planning injects a compact Evidence Choice reminder, execution injects Intent / Risk / Evidence, full TDD loads the slim skill, and hand-off injects tested/checked/skipped/residual-risk evidence rules.
- **Hook scripts (designing-tests Codex):** Direct Node.js (`codex-runtime.js`) with `user-prompt-submit` mode only. Designing-tests intentionally does not register SessionStart; UserPromptSubmit triggers only on explicit upstream `$superpowers:*` workflow skill mentions, including `$superpowers:finishing-a-development-branch`, and returns `{}` for unrelated natural language.
- **Markdown files:** Skills use YAML frontmatter with `name` and `description`. Knowledge base files use `last_updated` (YYYY-MM-DD), `updated_by`, `triggered_by_plan`.
- **`triggered_by_plan` rule:** Only update this field when a concrete plan filename can be identified as the trigger. If no plan triggered the update, **preserve the existing value — never overwrite with `null`**.
- **Content rules (KB):** `content-rules.md` is the shared SSOT for Project Knowledge Base maintenance and query routing. Defines language, inclusion/exclusion criteria, ownership matrix, feature capability reconciliation, quality standards, progressive shard layout, retrieval-cost guidance, query output fields, lightweight code-agent memory scope, Memory Candidate Gate, Diff Budget, ADR History Protection, and decision/glossary router rebuild compatibility.
- **JSON manifests:** `plugin.json`, native Codex hook files such as `codex-plugins/superpowers-memory/hooks/hooks.json`, Claude `hooks.json`, `marketplace.json`, and `codex-hooks-snippet.json` use 2-space indentation. Arrays/objects expand multi-line (one element per line) — both Claude and Codex tracks aligned. Codex command hooks set bounded `timeout` and omit optional `statusMessage` to keep the console quiet; native hook files and fallback snippets stay byte-shape aligned by release tests. `~/.codex/hooks.json` must remain strict JSON; never write JSON comments into it.
- **No linter configs present** — conventions followed by practice.

## Architecture Rules

- **Zero-modification principle:** Never modify upstream `superpowers` core files. Influence agent behavior through hook context injection and independent skills only (ADR-002).
- **Project-local knowledge base:** `docs/superpowers/memory/` lives in target project repos, not in this plugin repo. Plugins ship templates only and do not migrate retired memory directories (ADR-019, ADR-024).
- **No external dependencies beyond Node.js and git:** Hook scripts may only use tools present in standard Claude Code / Codex environments.
- **Cross-platform hooks:** Any new hook must work on Unix and Windows. The `run-hook.cmd` polyglot wrapper handles dispatch on Claude side. Codex side uses direct Node.js (no shell wrapper needed).
- **Strategy A for Codex track (ADR-013):** `codex-plugins/` is a parallel tree; never modify `plugins/` from Codex-side work. Designing-tests reference files are duplicated and must stay semantically aligned across tracks; release tests guard reference parity.
- **Architect design-pattern track parity:** Shared standards in `plugins/superpowers-architect/design-patterns/` and `codex-plugins/superpowers-architect/design-patterns/` should stay semantically aligned unless a change is intentionally host-specific. This restored v1.13.10 directory model includes bundled database and DDD/backend pattern files for the dynamic loader. `ddd-expert` remains separate and must not adopt a root `design-patterns/` directory or a catch-all `standards` skill.
- **DDD expert reference parity:** `plugins/ddd-expert/references/` and `codex-plugins/ddd-expert/references/` should stay semantically aligned. Modeling thought content belongs in `ddd-modeling-gates.md`; implementation/review risk routing belongs in `ddd-risk-router.md`; deeper rule content belongs in on-demand references. The DDD plugin must not grow a root `design-patterns` directory, hooks, or a catch-all `standards` skill.
- **DDD pattern ownership:** In `ddd-expert`, `$ddd-expert:domain-modeling` owns the evidence-first strategic modeling interview before design: inspect existing specs/glossary/ADRs/code/tests first, reconstruct user scenarios, event-storm in past-tense business facts, ask short one-question turns around authority/policy/hotspots, use `ddd-modeling-gates.md` for story/authority/lifecycle/invariant/failure/language/coordination thinking when boundaries matter, and synthesize conclusions only after a short question chain. `$ddd-expert:design` owns accepted-brief-to-implementable-design translation: confirm strategic model facts, answer relevant modeling gates before naming Aggregates, Repositories, ports, handlers, schemas, transactions, or event payloads, decide only implementation-relevant tactical mechanisms, name testing seams, and emit an Implementation handoff with accepted model source, capability/context/language, data authority, aggregate/policy/service/read-model boundary, modeling gate decisions, commands/queries/events/messages/reactions, consistency/failure boundary, and layer ownership; unknown material handoff facts are stop conditions. `$ddd-expert:implement` owns handoff-to-code translation: verify accepted source, handoff facts, and modeling evidence when boundaries matter, route confirmed object shapes and touched surfaces to the narrow references, translate placement/boundary/mechanism decisions to changed files by layer, report Rules Satisfied / Not Applicable / Exception, and stop back to domain-modeling/design/user when required facts are missing or when synchronous writes across candidate Aggregate Roots lack invariant/failure-tolerance evidence. `$ddd-expert:review` owns evidence-to-judgment: reconstruct expected model sources and model evidence, run the evidence gate across domain abstraction/spec behavior/code-level technology axes, classify DDD issues as modeling ambiguity/design violation/implementation placement error/allowed exception/evidence gap, and lead with findings or an explicit no-finding result plus residual gaps. `ddd-risk-router.md` owns shared implementation/review risk cards and explains how phases use them; its Routing Matrix names required references, required evidence, and allowed exceptions, and its probe commands are repo-shape-dependent examples, not fixed audit scripts. `ddd-modeling.md`, `ddd-core.md`, `ddd-agent-contract.md`, language guides, and Go support guides are on-demand rule sources loaded when a phase skill, risk card, task scope, or Architecture Gate requires them. Go guidance uses `github.com/go-jimu/components/ddd/event`, `github.com/go-jimu/components/ddd/message`, `github.com/go-jimu/contrib/message/kafka`, `github.com/go-jimu/components/taskqueue`, and `github.com/go-jimu/contrib/taskqueue/asynq` as the adopted component stack where the repository has opted in, including message.Runner, Kafka FailurePolicy, PeriodicTask, and PeriodicTaskScheduler.
- **DDD phase boundary rule:** Design is strategic-first and decides the model before code; tactical Aggregate/port/layer design follows only after subdomain, bounded context, language, data authority, lifecycle/invariant owner, failure tolerance, and context-map relationship are clear. Implement starts from an accepted model or explicit existing model and translates it into files, adapters, mappings, transactions, and tests; it must not invent model decisions or make multi-object synchronous writes prettier before challenging why the transaction exists. Review reconstructs the expected model from design/spec/code and applies risk-router evidence contracts; it reports violations, allowed exceptions, harmless local style, or evidence gaps, but does not redesign in review.
- **DDD responsibility-role routing rule:** `ddd-risk-router.md` classifies observed responsibilities before applying concept-name risk cards. It distinguishes same-BC reactions, cross-context contract consumers, boundary publishers, product read models, scheduled triggers, task processors, runtime loops, and Application coordination. Risk cards are for recurring boundary failure shapes, not one card per DDD concept name. The Manual Runner Misplacement card covers manual polling, reconciliation, scheduler, backlog drain, recovery, and outbox-drain loops: cadence/backoff/goroutine/shutdown policy belongs in taskqueue/runtime infrastructure unless a documented process-owned runtime exception exists, while business orchestration stays in Application services.
- **DDD Application port eligibility rule:** Technical capability classification and Domain mechanism placement precede interface ownership. Application-owned inward interfaces are normally QueryRepositories/read facades; command-side Application ports are exceptions that require a written gate showing why the need is not a Domain Repository, Aggregate method, Domain Service, Domain Event handler, Integration Message, named Application coordination service, ACL, or Infrastructure adapter. Generated proto structs are protocol DTOs/contracts, not Domain entities; Domain-facing ports use Domain types and map `Proto ↔ Domain` at Application/Interface/Infrastructure boundaries.
- **DDD generated adapter placement rule:** Generated IDL/RPC adapter placement is repository-calibrated. Physical Interface directories are optional package conventions, not mandatory layer names. When a generator already defines the external service/handler/controller contract, keep the hand-written adapter where the repository's language/framework convention owns that boundary, keep generated adapter methods thin, and do not create `interfaces/` solely because the generic layer diagram names an Interface layer.
- **DDD port granularity rule:** Port boundary is one stable semantic capability's full lifecycle (observe / mutate / publish / transfer / materialize / retire / release / recover), not a mechanism operation. Naming starts from a domain noun plus lifecycle role; verb-suffix names (`*Reader`, `*Writer`, `*Stream`, `*Opener`, `*Sender`, `*Fetcher`, `*Publisher`, `*Client`) require explicit justification of why the capability is one-directional. Inward-defined Port evolution defaults to adding a method to the existing Port when the new caller observes the same capability semantics; fork into a new Port only when the caller has different freshness, ordering, authorization, pagination, failure, consistency-window, published API, data-source, or test-substitute semantics, or operates on a different aggregate. QueryRepository specifically defaults to one port per bounded-context product read-model family, with new query scenarios added as methods rather than creating one interface per screen/RPC/handler. ISP/SRP at the port boundary segregate capabilities, not methods.
- **DDD technology-leak rule:** Do not create Application/Domain ports solely for MySQL transactions, broker publishing, retry counters, Unit of Work plumbing, cache/coordination routing read models, peer forwarding, network addresses, hop headers, retry/backoff, queue subjects, storage tables/keys, replica selection, or deployment topology. Hide those consistency and routing mechanics behind Repository, transaction-aware event bus, or Infrastructure adapters unless the use case names and observes that capability.
- **DDD CQRS and routing boundary rule:** CQRS query ports must answer application/product read use cases, not generic Infrastructure state lookup. Same read-store/projection source is a strong QueryRepository merge signal; different sources such as OLTP MySQL current-state reads versus ClickHouse analytics are split signals, but Application port names remain product-semantic rather than technology-semantic. `Peer`/`Directory`/`OwnershipDirectory` are illustrative names for distributed execution, worker, scheduler, and router systems: peer forwarding stays Infrastructure; `Directory.Get(workID) -> OwnerInfo{InstanceID, Addr}`-style owner-address lookup is Infrastructure routing; `OwnershipDirectory.AcquireLease/RefreshLease/ReleaseLease`-style lifecycle methods may be Application-owned only when they express active executor ownership semantics and exclude address lookup/forwarding responsibilities.
- **DDD Go infrastructure layout rule:** Go bounded-context `infrastructure/` stays flat by default. Primary adapter files and exported constructors use semantic capability names (`user_repository.go`, `NewUserRepository`, `runtime_state_redis.go`), while shared technology clients stay under `internal/pkg/<capability>`. Domain never imports `internal/pkg`; Application use cases use Domain/Application-owned ports or provider-neutral adopted component contracts rather than `internal/pkg` adapters; bounded-context Infrastructure receives initialized clients from `internal/pkg`.
- **DDD event / message boundary rule:** Domain Events are bounded-context-internal facts and the default home for repeated same-BC reactions to a state change; duplicate post-state-change side-effect handling should collapse into one same-BC Domain Event Handler. Cross-context state propagation uses Integration Messages with stable payload contracts translated by a Boundary Publisher from Domain Events or explicit published facts. Integration Message consumers live in separate Integration Message Handlers; `message.Subscriber` only registers handlers, while `message.Runner` lifecycle and Kafka FailurePolicy stay in Infrastructure/runtime. Keep inbound handlers thin; move multi-step orchestration into a named Application coordination service when needed.
- **DDD event collection drain ownership:** Each aggregate's event collection is drained exactly once by Application after a successful save (`Save()`, `save()`, or `repo.save()`). Repository never drains. Drain is one-shot; a second drain returns an empty collection, so callers must reload before further mutations instead of retrying persistence on a drained aggregate instance.
- **DDD Go logging rule:** Execution boundaries in generated Go guidance log one completion record for every success, failure, skip, or retry. Runtime logging uses `github.com/go-jimu/components/sloghelper`, with `sloghelper.Error(err)` for wrapped errors and request/job loggers passed through context or constructors.
- **DDD project-default Go component stack:** `ddd-golang.md` treats named Go libraries as the default stack for projects adopting this guide. Agents use public interfaces from `ddd/event`, `ddd/message` (including `message.Publisher`, `message.Handler`, `message.Subscriber`, and `message.Runner`), `contrib/message/kafka` (including explicit `FailurePolicy`), `components/taskqueue`, `contrib/taskqueue/asynq`, `fsm`, `sloghelper`, and `config` instead of inventing local substitutes unless existing repository code or explicit user direction establishes an exception; taskqueue scheduling uses `PeriodicTask`, `Schedule`, and `PeriodicTaskScheduler` from components rather than project-local scheduler abstractions.
- **DDD Go runtime config rule:** `ddd-golang-runtime.md` aligns with the adopted go-jimu config loader: component-owned `Option` structs, defaults.yml as the base file under service configs, explicit profile file prefix choices, `JIMU_PROFILES_ACTIVE` precedence, and placeholder-based env overrides.
- **DDD agent execution contract (ADR-015):** `ddd-agent-contract.md` is the on-demand agent-behavior layer for DDD work, not the default entrypoint. When loaded, agents classify the task as DDD-business / Go events/messages / Go taskqueue/polling/periodic / Go runtime-only / mixed, with an explicit manual polling/reconciliation runner route for `*_drain.go`, scheduler/recovery loops, timer/ticker goroutines, and `fx.Lifecycle` background loops. DDD/business portions emit the modeling §0 Architecture Gate, while pure runtime or `internal/pkg/taskqueue` wiring declares runtime/taskqueue component, module-assembly boundary, ownership, lifecycle, config, schema/processor/periodic registration, and shutdown impact instead. The contract leads with a hot-path Application command-side port decision card and P1-P7 self-checks (Port eligibility, Handler pressure, Read-side DTO, Event/message extraction, async role isolation, handler kind granularity, lightweight failure semantics), while preserving hard prohibitions including bloated Go fx entrypoints, manual runner as bounded-context root wiring, message runner/subscriber, and Kafka failure-policy substitutes; modeling/core/language docs retain short boundary reminders rather than duplicating the full agent protocol.
- **DDD Go runtime split (ADR-015):** Go runtime concerns — fx-based configuration management, fx module assembly guardrails, `fx.Lifecycle` hooks, graceful shutdown ordering, Kubernetes `preStop` — live in `ddd-golang-runtime.md`. `ddd-golang.md` keeps layers, aggregates, repositories, CQRS, file organization, and module assembly. Runtime examples catch unexpected server `Serve` errors and require shutdown dependencies to be explicit in constructors/invokes rather than assumed from conceptual architecture. In multi-service Go repos, service-specific runtime differences are expressed as named module variables such as `FooModule` / `BarModule` in a shared internal package module file; `cmd/<service>/main.go` loads its module instead of carrying provider lists.
- **DDD Go taskqueue split (ADR-017):** Taskqueue/polling/periodic concerns live in `ddd-golang-taskqueue.md`, not in the already-long Go guide. Agents adding polling/reconciliation jobs, periodic producers, `TaskType`, `PeriodicTask`, schema registry, task processors, asynq workers/schedulers, middleware, or `internal/pkg/taskqueue` must load the taskqueue guide; `ddd-golang.md` and `ddd-golang-runtime.md` only carry routing pointers. Periodic producers use the narrow `name + Schedule + Task + EnqueuePolicy` contract; config-disabled tasks are omitted from registration, `WithLocation` is cron-only, `WithProcessAt` / static `WithDeadline` are rejected for periodic policies, `WithUnique` is a short duplicate-suppression window rather than a period-length lock, `ErrDuplicatePeriodicTask` is scoped to one scheduler/registrar instance unless runtime coordination is documented, and business-visible scheduling/deadline/eligibility rules must be modeled outside provider mechanics.
- **DDD Go coding guardrails:** Go aggregate fields may stay exported for DTO/DO mapping, but business decisions and state changes must go through entity methods; transaction/session mechanics stay inside Infrastructure behind Repository/semantic methods; `application.go` RPC shortcut methods stay thin protocol adapters. Generated ConnectRPC/gRPC handler placement is repo-calibrated: when a repo already uses the Go shortcut, keep generated stubs in the Application package entrypoint file and do not create `interfaces/grpc`, `interfaces/connectrpc`, or `interfaces/runtime` packages solely to house generated RPC adapters.
- **DDD Go events/messages split (ADR-018):** Domain Event collection, same-BC event handlers, Boundary Publishers, Integration Messages, message handlers, message.Subscriber registration, message.Runner runtime-loop semantics, Kafka FailurePolicy, Kafka adapter wiring, idempotency, and event/message failure semantics live in `ddd-golang-events-messages.md`. Agents editing `application/eventhandler`, `application/messagepublisher`, `application/messagehandler`, `event.Collection`, `message.Publisher`, `message.Handler`, `message.Subscriber`, `message.Runner`, or Kafka FailurePolicy must load it; `ddd-golang.md` keeps only layer placement and routing pointers.

## Testing Conventions

- No automated test suite. Verification is done manually per plan task acceptance criteria.
- The `verify` command in `hook-runtime.js` / `codex-runtime.js` provides automated checks for KB files (stale path references, content-shape lint, ADR integrity, readiness, SSOT drift, retrieval-cost reporting, split candidates, advisory `coverageGaps`, decision/glossary router advisories, and strict `index.md` size). Codex variant runs the same `verify` logic.
- Fixture-based runtime testing: `plugins/superpowers-memory/hooks/fixtures/<scenario>/` (clean, dense-features, missing-feature-fields, shape-violation, ssot-violation, codex-apply-patch). `scripts/release/test/test_memory_verify.sh` copies fixtures to temporary non-repo directories so both `hook-runtime.js` (Claude) and `codex-runtime.js` (Codex) verify the same scenarios; Codex coverage includes canonical `PreToolUse` deny output for KB write protection, including multi-file `apply_patch` payloads where the memory target is not first.

## Git & Workflow

- **Commit message format:** `<type>: <description>` or `<type>(<scope>): <description>` (e.g., `feat(codex):`, `fix:`, `docs:`, `chore:`, `refactor:`, `style(codex):`).
- **Branch:** Feature work on `hotfix/<topic>` branches; merged to `main` via PR.
- **Versioning:** Bumped by `.github/workflows/auto-release.yml` after PR merge. `scripts/release/bump-versions.sh` always bumps `.claude-plugin/marketplace.json` metadata; changed Claude plugin paths bump Claude marketplace entries + `.claude-plugin/plugin.json`; changed Codex plugin paths bump `.codex-plugin/plugin.json`, native Codex hook files, and fallback `codex-hooks-snippet.json`. `test_codex_manifest_schema.sh` blocks deprecated `codex_hooks` README guidance, missing Codex hook `timeout`, noisy Codex hook `statusMessage`, and Claude runtime syntax in Codex plugin sources.
- **README synchronization:** Any change to `plugins/<name>/README.md` or `codex-plugins/<name>/README.md` must include a review of root `README.md`. Update root `README.md` when install flow, capabilities, marketplace schema, hook behavior, repository structure, release flow, or cross-track support changed; otherwise state that no root README update was needed in the PR or change summary.
- **Specs before plans:** Design specs (`docs/superpowers/specs/`), then implementation plans (`docs/superpowers/plans/`). Plans reference specs.
- **Plan checkboxes:** Implementation plan steps use `- [x]` / `- [ ]` syntax.

## Knowledge Base Content Rules (plugin-enforced)

- **Ownership matrix and Slot Contracts** — see `plugins/superpowers-memory/content-rules.md`. Each fact has ONE owner file; others reference by pointer (≤1 line). Each KB slot has owner intent, required shape, conditional shape, shard rule, excluded content, and verify coverage.
- **ADR granularity gate** — new ADRs only when a future reader without this record would re-propose the opposite (NORMAL 3-line default; CRITICAL only when ≥2 rejected alts with substantive analysis).
- **`features.md` is current capability map** — implemented capability groups follow `Product Capabilities`, `User / Operator Workflows`, `Platform Capabilities`, `Operations`; each `####` entry uses `Enables`, `Actors / Entry Points`, `Capability Boundary`, and `References`. No dense single-paragraph entries, missing implemented fields, commit SHAs, test counts, timestamps, or changelog narrative.
- **`glossary.md` is an alias router when large** — root entries stay ≤2 lines with one-line business definition + 1 path. Cross-context, ambiguous, renamed, or high-risk aliases stay in root; domain-local term clusters move to `glossary-<domain>.md`.
- **Exclusion Gate** in `ingest` checks every new entry against content-shape rules before write.
- **Progressive knowledge layout** — `index.md` is the only strict query-router file and must stay small enough for `query` to read first on demand. SessionStart reports KB availability/status but does not inline `index.md`. All other canonical entry files may split into `<slot>-<domain>.md` shards when a domain/submodule grows large. Large `decisions.md` roots become decision routers with stable families in `decisions-<domain>.md`; large `glossary.md` roots become alias routers with domain terms in `glossary-<domain>.md`. Do not delete valid knowledge merely to satisfy line or token pressure.
- **Ingest eligibility** — stale `covers_branch` is an inspection signal, not a write trigger. Incremental ingest runs only when changed source facts introduce or materially change durable project knowledge such as capabilities, architecture, conventions, dependency choices, decisions, glossary terms, lifecycle rules, or query answerability. Deployment-only, image/tag/version-only, formatting-only, and comment-only changes do not update `index.md` or `covers_branch` just to match HEAD.
- **`verify` surfaces** `ssotViolations`, `shapeViolations`, `readinessWarnings`, `retrievalCost`, `splitCandidates`, advisory `coverageGaps`, `qualityGate`, and `sizeWarnings` only for `index.md`. `qualityGate.ok` mirrors current `verify.ok`; `qualityGate.blockingFindings` counts stale refs, SSOT, shape, and readiness findings; `qualityGate.advisoryFindings` counts coverage, split, and size advisory findings. `retrievalCost`, `splitCandidates`, `decisions_family_shards_recommended`, and `glossary_alias_router_recommended` are advisory; `index_too_large` is a shape violation. `committable` reflects git state only.
- **KB writes go through `superpowers-memory:ingest` only** (ADR-010, ADR-024). PreToolUse hook blocks Write/Edit on `docs/superpowers/memory/` unless write-lock (`.git/superpowers-memory.lock`, 60-min TTL) is held. Codex write protection checks every `apply_patch` target, not just the first file. No escape hatch — manual typo fixes also go through `superpowers-memory:ingest`.
- **Schema meta-rule (slots vs topics)** — `content-rules.md` defines schema slots (file names, sections, ownership), not content topics. Concrete lists inside a slot (e.g., cross-cutting concern topics, implementation-constant examples) are discovery cues, not contracts; per-project content is discovered by reading the codebase. The former playbook lazy slot is removed and should not be regenerated.

## Codex-track-specific conventions (ADR-013)

- **Native Codex hook contract:** Each Codex plugin manifest declares its plugin-local native hook file. Native hook files use `{ "version": "<semver>", "hooks": { ... } }` and commands use `node "${PLUGIN_ROOT}/hooks/codex-runtime.js" ...` with bounded `timeout` and no `statusMessage`. Users need `[features] hooks = true` and `plugin_hooks = true`, then review/trust plugin hooks via `/hooks` and restart after install/upgrade.
- **Fallback hook snippet contract:** `codex-hooks-snippet.json` mirrors the native hook file for legacy fallback installer compatibility. Tests guard version/schema drift between manifest, native hook file, and fallback snippet while the legacy installer remains in tree.
- **Legacy hook installer protocol:** Each Codex plugin ships an installer; representative path: `codex-plugins/superpowers-memory/scripts/install-codex-hooks.js`. Public setup and cleanup skills are removed; the script remains for legacy migration tests and manual stale-hook removal. The installer supports `install` (legacy/private) and `remove`, prefers the native hook file, falls back to `codex-hooks-snippet.json`, infers the plugin name from source-tree and versioned cache layouts, removes stale entries for that plugin by runtime command path, writes strict JSON, and backs up `~/.codex/hooks.json`.
- **Fallback removal protocol:** There is no public `$<plugin>:cleanup` skill. Stale skill-workshop fallback commands in `~/.codex/hooks.json` are removed manually or by running `scripts/install-codex-hooks.js remove` from the installed plugin directory. The remove mode preserves unrelated hooks and deletes empty hook-event arrays.
- **Marketplace upgrade flow:** Codex `plugin marketplace upgrade` updates plugin files; native hooks take effect after restart. Current Codex users do not run setup after install or upgrade. Users with stale fallback entries remove them manually or run the script-only remove helper, then restart. README of each Codex plugin documents this.
- **Skill mention syntax:** Codex uses `$plugin:skill-name` (not `/`); UserPromptSubmit hook regex matches accordingly.
- **Architect prompt router:** Codex `superpowers-architect` registers SessionStart and UserPromptSubmit hooks from the restored v1.13.10 content. UserPromptSubmit must trigger only on explicit upstream `$superpowers:*` workflow skill mentions and must return `{}` for unrelated natural-language prompts. Codex `ddd-expert` remains hookless; DDD/backend activation there relies on explicit `$ddd-expert:design`, `$ddd-expert:implement`, or `$ddd-expert:review` invocation or host skill discovery from skill descriptions.
- **Designing-tests prompt router:** Codex designing-tests must not register SessionStart. UserPromptSubmit must stay non-blocking and trigger only on explicit upstream `superpowers` workflow skill mentions; natural-language prompts return `{}`. Injected content is the compact Evidence Choice primer plus the consolidated reference index, not the full skill.
- **Architect Stop policy:** Codex `superpowers-architect` does not install a Stop hook; legacy `stop` runtime mode is a compatibility no-op. Stop fires per assistant turn in Codex and is too intrusive. DDD guidance for `ddd-expert` relies on explicit `$ddd-expert:design` / `:implement` / `:review`, while architect standards can arrive from explicit upstream `$superpowers:*` workflow prompt mentions or `$superpowers-architect:standards`.
