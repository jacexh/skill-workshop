# Architecture

## System Overview

Skill Workshop is a dual-track plugin marketplace. Each track exposes companion plugins to a different host runtime: **Claude Code** via `plugins/` + `.claude-plugin/marketplace.json`, **Codex CLI** via `codex-plugins/` + `.agents/plugins/marketplace.json`. The active companion set includes `superpowers-memory`, `designing-tests`, explicit-only `superpowers-architect`, and DDD/backend-focused `superpowers-ddd-architect`. The two trees are independent (Strategy A — full duplication, ADR-013); the Codex track is experimental, and Claude remains the primary supported track. Each plugin is self-contained under its respective `<track>/<name>/` with hooks, skills, and templates.

## Data Flow

1. **Install (Claude):** `/plugin marketplace add jacexh/skill-workshop` → reads `.claude-plugin/marketplace.json` → user installs plugin → hooks auto-active.
2. **Install (Codex):** `codex plugin marketplace add jacexh/skill-workshop` → reads `.agents/plugins/marketplace.json` → user installs plugin → user enables `[features] hooks = true` and `plugin_hooks = true`, reviews/trusts plugin hooks via `/hooks` if needed, then restarts → Codex loads each manifest's native lifecycle hook file. No public setup skill is exposed.
3. **Codex upgrade flow:** `codex plugin marketplace upgrade` updates plugin files → user restarts Codex → native lifecycle hooks resolve the upgraded plugin root from the manifest. Current Codex users do not run setup after every upgrade; users with old fallback entries remove stale cache-path hooks manually or run `scripts/install-codex-hooks.js remove` from the installed plugin directory.
4. **Session start (Claude):** SessionStart hook → emits KB availability, `index.md` path, freshness status, and `superpowers-memory:query` guidance; it does not inline `index.md` content.
5. **Session start (Codex memory):** SessionStart hook → emits KB availability, `index.md` path, freshness status, and a two-rule standing primer. `query` reads `index.md` on demand. Standing primer compensates for absence of per-skill JIT (ADR-013, ADR-020).
6. **Knowledge management:** `superpowers-memory:rebuild` / `update` / `load` agent reads codebase / existing KB → writes/updates `docs/superpowers/memory/*.md`, including recognized non-index shard files such as `architecture-runtime.md`. Same skill content on both tracks.
7. **PreToolUse interception (Claude memory):** parses stdin → dispatches by `tool_name`. For `Skill`: per-skill advisory + `classifyFinishingState()` 4-way classifier when skill is `finishing-a-development-branch`. For `Write`/`Edit`/`MultiEdit`/`NotebookEdit` under `docs/superpowers/memory/`: blocks unless write-lock held (ADR-010).
8. **UserPromptExpansion interception (Claude memory):** matcher `finishing-a-development-branch` → same `classifyFinishingState()` classifier (ADR-012). Covers slash-typed path that bypasses `PreToolUse:Skill`.
9. **UserPromptSubmit interception (Codex memory):** regex on `prompt` field — `$superpowers:brainstorming` → load advisory; `$superpowers:finishing-a-development-branch` → KB-ready precheck → same `classifyFinishingState()` classifier reused with `eventName="UserPromptSubmit"` (ADR-013). Agent-self-decided invocation is uncoverable on Codex (documented gap).
10. **PreToolUse interception (Codex memory):** matcher `apply_patch|mcp__filesystem__.*` → `resolveTargetPath()` extracts path from `tool_input.{file_path,path,patch}` → returns `hookSpecificOutput.permissionDecision = "deny"` when path is under `docs/superpowers/memory/` and lock is not held (ADR-010 carries over; same lock file `.git/superpowers-memory.lock`).
11. **Skill interception (DDD architect Claude / designing-tests Claude):** DDD architect PreToolUse:Skill on selected workflow/review skills → injects DDD Design Guidance, DDD Implementation Guardrails, or DDD Boundary Review with `ddd-risk-router.md` first; design guidance is a compact reminder to start from Product semantics intake and Spec trace, while `$superpowers-ddd-architect:design` carries the full workflow. `superpowers-architect` no longer has automatic hook registrations and is explicit-only. Designing-tests PreToolUse injects test-design tier text.
12. **DDD architect / test guidance on Codex:** DDD architect uses lightweight SessionStart pointers, UserPromptSubmit matching for explicit upstream `superpowers` workflow skill mentions, and `$superpowers-ddd-architect:design` / `:implement` / `:review` for full on-demand DDD/backend guidance; design-phase UserPromptSubmit guidance names Product semantics intake, Spec trace, and command/query/event/lifecycle modeling without inlining the full method. Natural-language DDD discussion stays quiet unless the skill is invoked. `superpowers-architect` is explicit-only. Designing-tests intentionally does not register SessionStart; it reinforces architecture-aware test design and hand-off evidence only through UserPromptSubmit when users explicitly mention upstream workflow skills. No implicit agent-self-decided skill-call JIT is available on Codex (ADR-013, ADR-014, ADR-021, ADR-022).
13. **Auto release:** PR merge to `main` → `.github/workflows/auto-release.yml` computes the next repository tag → detects changed plugin paths independently for Claude and Codex tracks → `scripts/release/bump-versions.sh` updates the relevant manifests plus native/legacy Codex hook configs → workflow commits the bump, tags it, and creates a GitHub Release.

## Key Design Decisions

- **Zero-modification principle:** Plugins never modify upstream `superpowers` files — behavior influence via hook context injection only (ADR-002).
- **Project-local knowledge base:** `docs/superpowers/memory/` lives inside the target project repo, versioned alongside code (ADR-003, ADR-019). Legacy `docs/project-knowledge/` is hard-migrated with `git mv` and is not a normal read/write path.
- **Knowledge split into separate files:** Enables surgical incremental updates (ADR-003).
- **Progressive KB layout:** `index.md` remains the only strict query-router size constraint; all other entry files may split vertically into `<slot>-<domain>.md` shards, and retrieval cost is advisory rather than a reason to delete valid knowledge (ADR-016, ADR-020).
- **PreToolUse/query over SessionStart for KB injection (Claude):** Precise skill-time guidance and explicit `query` reads maximize compliance while SessionStart stays lightweight (ADR-004, ADR-020).
- **Index-first progressive loading:** `query` reads `index.md` first and then loads owner files/shards on demand; SessionStart only announces the KB and status (ADR-005, ADR-020).
- **Node.js hook runtime:** Single `hook-runtime.js` for all superpowers-memory hooks (ADR-007).
- **Evidence-based staleness (Claude):** Now via per-skill PreToolUse advisories pointing to `load`/`update`; the previous Stop-hook-based detection was removed in commit `e6153b8` because per-turn firing produced noise rather than signal.
- **KB write-lock:** PreToolUse blocks edits to `docs/superpowers/memory/` and legacy `docs/project-knowledge/` unless `ingest` or compatibility aliases hold the lock (ADR-010, ADR-019).
- **Rich injection over hard block for staleness:** Finishing-branch hook returns architect-style rich context (diff scope + imperative MUST) instead of `decision: block`; hard block reserved for catastrophic KB-missing (ADR-011).
- **Dual-path coverage of finishing-a-development-branch (Claude):** PreToolUse:Skill (programmatic) and UserPromptExpansion (slash-command) share `classifyFinishingState()`; KB-ready precheck at each caller (ADR-012).
- **Codex marketplace compat — Strategy A:** Parallel `codex-plugins/` tree with full duplication; Codex track is experimental. Native manifest hooks are the install/upgrade path; public setup and cleanup skills are removed after native hooks became primary, while script-only fallback removal remains for stale `~/.codex/hooks.json` entries (ADR-013, ADR-014). DDD architect Codex layers lightweight SessionStart pointers, explicit workflow-skill prompt routing, and explicit `design` / `implement` / `review` skills; Stop interception is intentionally not registered. Codex still cannot JIT-hook native skill invocation, so skill-specific Claude behavior is approximated with lightweight SessionStart/UserPromptSubmit only.
- **DDD architect plugin split:** `superpowers-ddd-architect` is the active DDD/backend architecture guardrail plugin on both tracks. It uses shared references under `plugins/superpowers-ddd-architect/references/` and `codex-plugins/superpowers-ddd-architect/references/`, reads `ddd-risk-router.md` first, treats search probes as repo-calibrated examples, and avoids the legacy root `design-patterns/` dynamic loader. `superpowers-architect` is retained as explicit-only general standards lookup without bundled DDD/database defaults (ADR-021, ADR-022, ADR-023).
- **PR-merge auto release:** Release logic is script-backed; Claude and Codex plugin version bumps are path-scoped and Codex hook snippet versions stay aligned with Codex plugin manifests (see `docs/superpowers/specs/2026-04-27-auto-release-versioning-design.md`).
- **DDD agent contract + language/runtime splits:** Agent-behavior contract for DDD work, Go runtime concerns, Go events/messages concerns, Go taskqueue/polling/periodic concerns, plus Go/Python/TypeScript language guides live in standalone DDD reference files, loaded by `superpowers-ddd-architect` phase-specific skills alongside modeling/core guidance (ADR-015, ADR-017, ADR-018, ADR-022, ADR-023).
